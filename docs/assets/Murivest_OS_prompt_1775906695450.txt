📘 DOCUMENT 5: DATABASE DESIGN DOCUMENT / ERD SPECIFICATION
Project: Murivest OS
Document Type: Database Design & Entity Relationship Specification
Purpose: Define the database structure, schema grouping, key relationships, indexing logic, and data governance rules for Murivest OS.

1. Purpose of the Database Design
This document defines how Murivest OS data should be structured so that:

all departments operate from one connected data architecture

no important data is duplicated unnecessarily

permissions can be enforced cleanly

reporting remains accurate

portals can expose filtered data safely

workflows can move records between departments

the system can scale without becoming disorganized

This is the technical data backbone of Murivest OS.

2. Database Design Principles
Murivest OS database design should follow these principles:

2.1 Single Source of Truth
Core entities such as contacts, properties, mandates, deals, campaigns, users, and documents should exist once and be reused across modules.

2.2 Domain Separation
The database should be organized by domains, not by random tables.

2.3 Controlled Redundancy
Some summary tables and reporting tables may duplicate data intentionally for analytics and speed, but operational truth must remain centralized.

2.4 Strong Relationships
Foreign keys and constrained relationships must enforce data integrity.

2.5 Auditability
Critical records must retain history of changes.

2.6 Portal Safety
Investor, landlord, and tenant portals must read from approved filtered views, not raw unrestricted internal tables.

2.7 Scalability
The schema must support larger property inventories, more deals, more investors, more campaigns, and larger training content libraries.

3. Recommended Database Architecture
Murivest OS should use:

A. Primary Relational Database
For core operational records.

B. File/Object Storage
For documents, brochures, media, valuations, signed agreements, and manuals.

C. Search Index
For fast cross-platform search.

D. Reporting Warehouse / Analytics Store
For dashboards, KPIs, trend analysis, and campaign metrics.

E. Audit/Event Store
For immutable operational and security logs.

Layer 1: Role-Based Access Control (RBAC)
Every user belongs to one or more roles.

Layer 2: Permission-Based Action Control
Roles map to exact permissions such as:

view

create

edit

delete

approve

publish

export

assign

close

grant access

Layer 3: Record-Level Access Control
Even if a role can view “contacts,” it may only view:

assigned contacts

department contacts

approved contacts

self-owned portal records

Layer 4: Field-Level Visibility
Some users may view a record but not certain sensitive fields.

Example:
A Sales Executive may view a property, but not:

commission split

legal risk memo

internal valuation commentary

Layer 5: Workflow Authority
Some users may start a workflow, but only others may approve it.
